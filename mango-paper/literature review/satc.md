# [SatC](https://www.usenix.org/system/files/sec21fall-chen-libo.pdf)
 
### __Overview__
- Our key insight is that, string literals on web interfaces are commonly shared between front-end files and back-end binaries to encode user input
- We implemented a prototype of SaTC and evaluated it on 39 embedded system firmwares from six popular vendors. SaTC discovered 33 unknown bugs, of which 30 are confirmed by CVE/CNVD/PSV.
- Compared to the state-of-the-art tool KARONTE, SaTC found significantly more bugs on the test set.
- CWEs
    - Memory Corruption
    - Command Injection


### __Listed Contributions__
- We propose a novel technique that leverages the common
keywords between the front-end and back-end of embedded
systems to locate the data entry in the back-end binary.
- We design and implement SaTC that utilizes coarse-grained
taint analysis and trace merging method to efficiently detect
vulnerabilities in embedded systems.
- We evaluate SaTC on 39 real-world firmware samples and
discover 33 unknown bugs, including command injection,
buffer overflow, and incorrect access control bugs.

### __Problem__
- Threat Model: Routers and Cameras
- Challenges:
    - C1: Identifying keywords in the front-end
    - C2: Locating the input handler in the back-end
    - C3: Tracking the massive paths of user input to detect vulnerabilities

### __Input Keyword Extraction__
- Extracts keywords from front-end files.
- Classified into parameter or action keywords
- HTML and XML use standard formats JS is turned into an AST and parsed
- discard strings with special characters and remove =
- __Border Binary Recognition__: Just callsites with matched front-end strings

### __Input Entry Recognition__
- Pairs front-end string with back-end location
- Considered a match if any backend-string startswith any front-end string
- Attempts to rectify missing front-end keyword for backend-keyword by recognizing similar back-end function calls
- Supports NVRAM and ENV Variables for their CPEF (Cross Process Entry Finder)

### __Input Sensitive Taint Analysis__
- Coarse-Grained Taint Engine: Lightweight data-flow analysis
    - Taint source should be related to user input
    - Balance of accuracy and efficiency
    - Only tracks the data flow from source to potential sink
    - Propagates attributes at instruction level
- Separates functions into Summarizable, general, and nested functions

### __Efficient Path Exploration__
- detects functions semantically equivalent to memcpy (e.g. strcpy)
- Sensitive Trace Guiding
    - Searches the sink call traces for each target before exploring a path, deleting entries from unreachable functions
- Call Trace Merging
    - SaTC merges call traces with the same input entry as much as possible.

### __Evaluation__
- Dataset
    - 39 firmware samples from 14 series
    - 37 routers and 2 cameras

- Existing Tool
    - Compared against KARONTE

- Bug Confirmation
    - If path is reachable in back-end from front-end and can manually generate PoC then it's a real bug

### __Real-world Vulnerabilities__
- Shows list of 33 bugs generated by SaTC
- Compares results with KARONTE on KARONTE Dataset
    - SaTC: 2084 alerts, 683 TP -> 32.77% TP to FP
    - KARONTE: 74 alerts, 46 TP -> 62.16% TP to FP

- Compares against KARONTE on 7 of their firmware
    - KARONTE finds 0 TP
    - SaTC find 36 TP

### __Limitations__
- Suffers from over-taint problems due to missing function summaries
- High amount of Alerts and low TP rate (33%)
- As slow as Karonte
- No path verification

### __Strengths__
- Well deveolped Border Binary identification
- Finds higher likely entry-points compared to related work
