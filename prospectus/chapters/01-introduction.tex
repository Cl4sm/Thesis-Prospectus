\chapter{Introduction}
\label{ch:intro}

\section{The Problem}

In the Netgear R6400 router, a command injection vulnerability sits in a binary called \texttt{dlnad}. The attack path is short: a user-controlled parameter named \texttt{iserver\_passcode} flows through the \texttt{httpd} web server into NVRAM, where \texttt{dlnad} reads it back out and passes it to \texttt{system()} without sanitization. SaTC~\cite{chen2021satc}, the state of the art in firmware taint analysis at the time, never finds this bug. Neither does Karonte~\cite{redini2020karonte}. Both tools restrict their analysis to ``border binaries'' (binaries they determine are likely to process network input), and \texttt{dlnad}, with its small size and absence of frontend keyword matches, falls outside that scope. The vulnerability is real, it is exploitable, and the best tools miss it because they filter away the binary that contains it.

This is a detection failure, and it is common. IoT firmware runs on roughly 15 billion devices worldwide as of 2024~\cite{statista2024iot}, with projections exceeding 30 billion by the end of the decade. Most of that firmware is written in C, compiled for MIPS or ARM, and ships without stack canaries, without ASLR, and without anyone having run a static analyzer on it. The Mirai botnet~\cite{antonakakis2017mirai} showed the consequences: hundreds of thousands of IoT devices conscripted into a DDoS network that brought down major infrastructure. A single vendor ships hundreds of firmware builds across its product line. Manual auditing cannot keep pace.

But detection is only half the problem. Suppose a tool could scan every binary in every firmware image and flag every potential vulnerability. The result would be thousands of candidates, and a candidate is not a confirmed vulnerability. Static analysis produces false positives. Fuzzers produce crashes that need diagnosis. Pattern matchers flag code that looks dangerous but may be safe. Each candidate requires a human analyst to determine whether the bug is real, whether it is exploitable, and how it should be fixed. Skilled security analysts are scarce, and this validation bottleneck is the real constraint on how much security analysis gets done.

The same tension appears at the source level. Codebases like SQLite, the Linux kernel, and Jenkins contain millions of lines of code and evolve constantly. DARPA's AI Cyber Challenge (AIxCC)~\cite{darpa2025aixcc}, a \$29.5 million two-year competition, forced teams to build systems that find, validate, and patch vulnerabilities in these projects with zero human intervention. The competition revealed genuine progress but also exposed how far current tools fall short (considerably). Autonomous systems can detect some bugs; reliably confirming that a detection is a real, exploitable vulnerability, and then fixing it, remains hard.

Whether the target is a stripped MIPS binary from a router's firmware or a C project in Google's OSS-Fuzz corpus, automated vulnerability discovery runs into the same bottleneck: the gap between detecting a potential vulnerability and confirming it is real. This dissertation attacks that gap.


\section{The Detection-Validation Gap}

The gap between detection and validation manifests differently at different levels of analysis, but the underlying pattern is consistent: detection scales, validation does not.

\textbf{At the static analysis level,} taint analysis can flag thousands of potential data flows from sources to sinks across an entire firmware corpus. But a taint hit is weak evidence. Traditional taint analysis tracks a single bit per value (tainted or not), which tells you that untrusted data reached a dangerous function. It does not tell you whether that data is still in a form that could trigger the vulnerability. A value that has been converted to an integer by \texttt{itoa()} before reaching \texttt{system()} is not exploitable, but binary taint analysis flags it anyway. At scale, this imprecision produces thousands of candidates that bury the real bugs. The analyst (human or automated) must perform the validation work that the detector left undone.

\textbf{At the system level,} autonomous Cyber Reasoning Systems can orchestrate multiple detection techniques: coverage-guided fuzzing, grammar-based fuzzing, LLM-powered code scanning, and more. Each technique produces its own form of evidence. A fuzzer produces a crash. An LLM scanner flags a suspicious pattern. A static analyzer identifies a data flow. Individually, each piece of evidence is inconclusive. Validation at this level means building convergent evidence: when multiple independent techniques point to the same bug, and a patch fixes it without breaking functionality, the case becomes strong. But orchestrating this convergence across heterogeneous tools, autonomously, remains an open problem. DARPA's Cyber Grand Challenge~\cite{darpa2016cgc} demonstrated the first generation of autonomous CRS on purpose-built binaries in 2016; AIxCC raised the bar to real software, and the jump is enormous.

\textbf{At the formal analysis level,} symbolic execution~\cite{king1976symbolic} can generate concrete inputs that trigger a specific program path, providing the strongest form of automated validation short of a working exploit. But path explosion~\cite{krishnamoorthy2010tackling} makes symbolic execution impractical on full programs. The key question is whether targeted symbolic execution, guided by static analysis findings and operating on an intermediate representation that carries recovered type information, can make formal validation tractable for specific vulnerability candidates. Binary analysis frameworks like angr~\cite{shoshitaishvili2016sok} already recover structured control flow and type information through their decompilation pipelines. A symbolic executor operating on this recovered structure could generate tighter constraints and explore fewer spurious paths. But no such engine exists yet.


\section{Thesis Statement}

This dissertation argues that the gap between automated vulnerability detection and confirmed exploitation can be closed through progressively stronger forms of validation evidence: value-level reasoning that filters implausible candidates during static analysis, multi-technique convergence that builds confidence through independent corroboration, and symbolic feasibility analysis that produces concrete triggering inputs.


\section{Contributions}

The 3 projects in this dissertation each contribute a different form of validation evidence. Each project's limitations reveal what the next form of evidence must provide.

\textbf{Operation Mango} (Chapter~\ref{ch:mango}) bakes partial validation into the detection phase itself. Published at USENIX Security 2024~\cite{gibbs2024mango}, Mango introduces MangoDFA, a novel static data-flow analysis engine for finding taint-style vulnerabilities in firmware binaries. Where traditional taint analysis tracks binary taint bits (``tainted or not''), MangoDFA tracks the full transformation history of values through Rich Expressions, which encode how a value was constructed as it flows through the program. A Rich Expression reaching \texttt{system()} carries evidence about whether the value is still exploitable: was it constrained by \texttt{itoa()}? Truncated by \texttt{snprintf()}? Concatenated raw into a shell command? This is partial validation embedded in the analysis itself. Combined with backward tracing from sinks (Sink-to-Source) and selective callee skipping (Assumed Nonimpact), Mango analyzes \textit{every} binary in a firmware image. On the standard Karonte dataset, Mango analyzed all 3,599 binaries (versus SaTC's 131), a 27$\times$ increase in scope, while averaging 8 minutes per binary compared to SaTC's 6.5 hours. On a large-scale corpus of 1,698 firmware images (770,374 binaries), Mango produced 10,834 ranked vulnerability alerts and confirmed 166 exploitable vulnerabilities in a sampled audit. Mango's limitation: Rich Expressions track value transformations but cannot determine whether the specific branch sequence required to reach a sink is actually satisfiable. The result is candidates with strong evidence attached, but no formal proof of feasibility.

\textbf{Artiphishell} (Chapter~\ref{ch:artiphishell}) validates through multi-technique convergence. Built for the DARPA AI Cyber Challenge~\cite{darpa2025aixcc} and designed and led by the author at ASU's SEFCOM lab, Artiphishell is a fully autonomous Cyber Reasoning System. The system bolts together coverage-guided fuzzing, grammar-based fuzzing, LLM-powered vulnerability scanning, statistical root-cause analysis, and AI-driven patching into a multi-agent pipeline. All agents share state through the Analysis Graph, enabling coordination: when a fuzzer crashes on a function that the LLM scanner also flagged, and root-cause analysis localizes the bug, and a patch fixes it without breaking other tests, the system has built a multi-source case for the vulnerability. Artiphishell advanced to the AIxCC semi-finals and was presented at DEF CON 2025. The central lesson: convergent evidence from independent techniques provides strong validation, but precision on complex candidates remains constrained by the power of the underlying analyses.

\textbf{Symbolic Execution on AIL} (Chapter~\ref{ch:proposed}) provides formal validation. This proposed work builds a symbolic execution engine that operates on angr's AIL intermediate language rather than the lower-level VEX IR. AIL carries recovered type information and structured control flow, enabling tighter SMT constraints and fewer infeasible paths. The engine plugs directly into MangoDFA as a second-phase validator: Mango identifies candidates with partial evidence (Rich Expressions), and the AIL symbolic executor performs targeted exploration of those specific paths to determine feasibility and generate concrete triggering inputs. A successful validation produces an input that triggers the vulnerability. A failed validation proves the path infeasible, eliminating a false positive. A third research thrust uses coverage data from Artiphishell-style fuzzing to guide the symbolic explorer's path prioritization, closing the loop between all 3 projects.
