\chapter{Proposed Work: Automated Validation of Static Vulnerability Candidates Through Directed Symbolic Execution}
\label{ch:proposed}

\section{Problem and Approach}
\label{sec:proposed-approach}

Static analysis tools such as \mango (Chapter~\ref{ch:mango}) produce large numbers of vulnerability candidates across a firmware image, but every candidate remains a hypothesis with no validation.
Necessarily, these approaches approximate program behavior which may result in false positives: attacker input flowing into \texttt{system()} may be blocked by an NVRAM flag check, a string comparison against a hardcoded whitelist, or a cross-binary precondition that static analysis cannot resolve.
Vendors routinely reject reports that lack proof-of-concept exploits~\cite{gibbs2024mango}, and manual triaging multiple firmware is infeasible.

Symbolic execution~\cite{king1976symbolic} can close this gap by producing concrete triggering inputs or proving path feasibility.
Whole-program symbolic execution, however, is intractable for firmware services due to path explosion~\cite{krishnamoorthy2010tackling}, unmodeled environmental state (NVRAM, HTTP APIs, cross-binary IPC)~\cite{greenhouse}, and the verbose constraints produced by low-level IRs such as VEX~\cite{nethercote2007valgrind}.
Directed approaches~\cite{ma2011directed,vadayath2022arbiter} reduce explosion by targeting specific locations, but still face the entry-point problem of complex environmental preconditions that must be satisfied when analyzing firmware.

Static analysis systems that identify source-to-sink paths already solve the expensive sub-problem of determining which paths to examine.
I propose a directed symbolic execution system that consumes these paths and collapses the validation problem from open-ended exploration into bounded trace verification.
The system accepts traces from an upstream static analysis tool, normalizes them into a canonical representation encoding the function-level path from source to sink, and produces for each trace a verdict (reachable, unreachable, or constant sink) along with concrete inputs for confirmed paths.
The normalization layer accepts any taint-style analysis that produces source-to-sink call chains.

Three design choices distinguish this system from conventional binary symbolic execution.
First, execution operates on angr's AIL (angr Intermediate Language), a decompiler-level IR that carries recovered type information and structured control flow, producing tighter constraints and fewer lifting artifacts than VEX-based execution.
Second, a hybrid concretization policy combines static precondition extraction (NVRAM keys, HTTP parameters, guard patterns from decompilation) with LLM-directed precondition recovery for cases where static extraction is insufficient.
This policy allows guard checks to resolve deterministically while preserving provenance annotations that track which attacker-controlled sources influence the sink argument.
Third, for vulnerabilities spanning multiple binaries through shared NVRAM state, the system traces SET and GET locations across the firmware image, constructing multi-stage constraint chains that connect the setter binary's symbolic state to the getter binary's path conditions.

\section{Planned Evaluation}
\label{sec:proposed-evaluation}

The proposed evaluation will expand the prototype across multiple firmware images, vendors, and architectures to assess generalization.
I plan to conduct the following experiments:

\begin{enumerate}[nosep]
  \item \textbf{Multi-firmware effectiveness.} Evaluate the system on firmware images spanning multiple vendors and architectures to show generalizability. This includes the reporting of 0-day vulnerabilities discovered through this process (of which we have identified one already).
  \item \textbf{Static analysis verification.} Evaluate the system's effectiveness on multiple static analysis tools (e.g., \mango, \satc, \karonte, etc.) to demonstrate the normalization layer's generality and identify the gap between reported candidates and confirmed vulnerabilities.
  \item \textbf{LLM-directed precondition recovery.} Evaluate the contribution of LLM-directed precondition recovery by comparing validation results with and without this component.
\end{enumerate}
